# 提示工程

提示工程（Prompt Engineering）是设计和优化大语言模型输入提示的技术，以引导模型产生更准确、更有用的输出。在 LLM 时代，**提示就是编程**。

## 基本原则

### 1. 清晰明确

❌ "告诉我关于 AI 的事"

✅ "用 3-5 个要点，解释 2024 年人工智能领域最重要的 3 个技术突破，每个突破包含：名称、原理简述、实际应用案例"

### 2. 提供上下文

告诉模型它的角色、目标受众和输出格式。

### 3. 给出示例

通过示例展示期望的输入-输出模式，比纯文字描述更有效。

### 4. 逐步推理

对于复杂任务，引导模型分步思考而非直接给出答案。

### 5. 约束与否定

告诉模型什么**不要**做，和告诉它做什么一样重要：

```
用 200 字以内解释量子计算，要求：
- 不要使用数学公式
- 不要使用"简单来说"这类填充词
- 面向高中生水平
```

## 核心技巧

### Zero-shot 提示

直接给出指令，不提供示例：

```
将以下文本分类为正面、中性或负面情感：

文本：这款产品的用户体验非常流畅，但价格偏高。
情感：
```

### Few-shot 提示

提供少量示例帮助模型理解任务模式：

```
将评论分类为正面或负面：

评论：服务态度很好 → 正面
评论：等待时间太长了 → 负面
评论：菜品味道不错但环境一般 →
```

示例数量与选择策略：

| 类型 | 示例数 | 适用场景 |
|------|--------|---------|
| Zero-shot | 0 | 简单任务 |
| One-shot | 1 | 格式化任务 |
| Few-shot | 2-5 | 复杂模式学习 |

**示例选择策略**：优先选择与当前输入语义相似的示例（基于嵌入向量检索），确保示例覆盖不同模式类型，从简单到复杂排列。

### 角色扮演

赋予模型一个专业角色，让输出更专业：

```
你是一位资深的机器学习工程师，擅长将复杂概念解释给初学者。
请解释什么是梯度下降算法，使用生活中的类比。
```

### 输出格式控制

明确指定输出格式，便于程序解析：

```
请分析以下代码的时间复杂度，用以下 JSON 格式输出：
{
  "complexity": "O(?)",
  "explanation": "一句话解释",
  "bottleneck": "瓶颈所在行"
}
```

## Chain-of-Thought 推理

CoT 让模型通过逐步推理解决复杂问题，而非直接猜测答案。

### 基础用法

**隐式触发** — 在提示末尾添加"让我们一步步思考"：

```
如果小明有 5 个苹果，给了小红 2 个，又买了 3 个，请问小明现在有几个苹果？

让我们一步步思考。
```

**显式示例** — 提供带推理过程的示例：

```
Q: 停车场有 3 辆车，又来了 2 辆，开走了 1 辆，还有几辆？
A: 停车场原有 3 辆车。又来了 2 辆，所以 3 + 2 = 5 辆。开走了 1 辆，所以 5 - 1 = 4 辆。答案是 4。

Q: 图书馆有 15 本书，借出 8 本，还回 3 本，还有几本？
A:
```

### 高级变体

- **Self-Consistency**：多次采样推理路径，取多数投票结果
- **Auto-CoT**：让模型自动生成推理示例，省去手动编写
- **Tree-of-Thought（ToT）**：将推理组织为树结构，允许探索和回溯

CoT 对数学推理、逻辑推理、多步骤规划特别有效。注意 CoT 会增加输出 token 数，简单任务无需使用。

## 提示 vs 微调

| 维度 | 提示工程 | 微调 |
|------|---------|------|
| 成本 | 低（仅推理） | 高（需要训练） |
| 灵活性 | 高（随时修改） | 低（需重新训练） |
| 效果上限 | 中等 | 高 |
| 知识更新 | 实时 | 需重新训练 |
| 适用场景 | 快速原型、通用任务 | 专用领域、极致性能 |

## 迭代优化

提示工程是迭代过程：编写初始提示 → 评估输出 → 分析不足 → 修改提示 → 重复直到满意。
